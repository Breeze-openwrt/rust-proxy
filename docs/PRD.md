# 🚀 Rust-Proxy 产品需求文档 (PRD)

> **版本**: v1.0  
> **状态**: 激活 (Active)  
> **核心愿景**: 打造全球最快、最轻、基于内核加速的 SNI 自适应代理引擎。

---

## 1. 业务背景 (Background)
在处理 TB 级的高并发 HTTPS 流量时，传统用户态代理（如 Nginx、HaProxy）在处理加解密握手和跨内核数据拷贝时存在明显的性能瓶颈。本项目旨在使用 Rust 结合 Linux 现代内核特性，彻底打破这些枷锁。

## 2. 目标用户 (Target Users)
- **高性能网关管理员**: 需要在单机处理万级并发连接的站点。
- **隐私保护倡导者**: 需要在不解密 TLS 报文的前提下，根据 SNI 域名进行智能分流。
- **暴力下载/上传爱好者**: 追求带宽物理限制下的极致吞吐表现。

## 3. 核心功能需求 (Functional Requirements)

### 3.1 SNI 智能嗅探
- **需求**: 在 TLS 握手的第一阶段 (Client Hello) 准确提取目标主机名 (Server Name Indication)。
- **约束**: 禁止对流量进行中间人解密 (MITM)，保持数据的原始安全性。

### 3.2 暴力流量中继
- **需求**: 支持多核心 Accept，实现负载均衡。
- **加速**: 应用 `io_uring` 消除用户态与内核态的上下文切换。

### 3.3 内核态重定向
- **需求**: 对于已鉴权的持久连接，将转发逻辑下沉至 eBPF 模块。
- **效果**: 实现数据包在内核空间的“原地瞬移”，完全绕过用户态内存拷贝。

## 4. 非功能性需求 (Non-Functional Requirements)

### 4.1 极致性能 (Performance)
- **吞吐量**: 在 Debian amd64 (v3) 架构下，吞吐量应接近物理网卡上限。
- **响应时延**: 握手解析耗时应控制在微秒 (μs) 级别。

### 4.2 极高稳定性 (Stability)
- **资源利用**: 采用 Thread-per-core 模型，杜绝线程间竞争。
- **容错**: eBPF 模块故障时应能平滑降级到用户态，业务不中断。
